# Резервный механизм для ML-рекомендаций в TravelTech

## 🎯 Проблема
ML-модель рекомендаций может:
- Полностью отказать (ошибка сервиса)
- Отвечать дольше 3 секунд
- Возвращать рекомендации с низким confidence

**Задача:** Обеспечить бесперебойную работу сервиса без кэширования предсказаний.

---

## 🏗 Архитектура решения

# Запасной план (Fallback) для ML-рекомендаций

## В чем проблема
Наша ML-модель, которая советует отели и туры, может сломаться, зависнуть или выдавать странные результаты. 
Показывать пользователю ошибку 500 или пустой экран — плохо. Кэшировать ответы ML тоже нельзя, потому что интересы людей быстро меняются (сегодня ищет пляж, завтра — горы).

## Моя идея решения
Сделать прослойку на бэкенде. Фронтенд просто запрашивает рекомендации, а бэкенд сам решает, где их взять. Если ML работает нормально — берем у него. Если ML «тупит» или упал — незаметно для юзера подсовываем запасные варианты из нашей базы.

## Логика запасных вариантов (что показывать)
Если ML не ответил, идем по шагам:

1. **Смотрим историю пользователя.** 
   Берем из PostgreSQL его последние бронирования. Если он часто брал отели 4 звезды у моря, показываем ему похожие из нашей базы, отсортированные по рейтингу.
2. **Смотрим профиль.** 
   Если истории нет, берем данные из профиля (бюджет, путешествует с детьми или один). Фильтруем базу по этим тегам.
3. **Показываем популярное.** 
   Если это совсем новый пользователь (нет ни истории, ни заполненного профиля), просто отдаем Топ-10 самых популярных отелей. Этот список будем хранить в Redis и обновлять раз в час через простой скрипт по расписанию.

## Как переключаемся (Таймауты и Circuit Breaker)
Чтобы не заставлять пользователя ждать 10 секунд, пока ML думает:
* Ставим жесткий **таймаут 3 секунды**. Если за 3 секунды нет ответа — сразу идем в Fallback.
* Чтобы не «добивать» упавший ML-сервис тысячами запросов, добавим простую библиотеку **Circuit Breaker** (например, `pybreaker` для Python). 
* *Как это работает:* если ML вернул ошибку 5 раз подряд, «предохранитель» срабатывает, и мы следующие 5 минут вообще не дергаем ML, а сразу идем в запасной вариант. Потом пробуем снова.

## Где что лежит
* **PostgreSQL:** история бронирований и профили юзеров.
* **Redis:** заранее посчитанный топ популярных отелей (чтобы отдавать моментально).
* **Бэкенд (Python):** вся логика переключения, таймауты и Circuit Breaker.

## Как будем мониторить
Чтобы не узнать о поломке от пользователей, нужно следить за метриками:
* Считать, сколько раз сработал фоллбэк. Если больше 20% запросов идут через запасной план — шлем алерт в Telegram/Slack.
* Смотреть на время ответа ML (строить графики).
* Логировать причины переключения (таймаут, ошибка 500 или низкий confidence).

## О чем еще подумал (Edge cases)
* *Что если Redis тоже упадет?* На этот случай можно захардкодить небольшой статический список «лучших отелей» прямо в конфиге, чтобы хоть что-то показать.
* *Что если ML вернул ответ, но он плохой?* ML-модель обычно отдает скор (уверенность). Если скор ниже 0.7, мы считаем это как ошибку и тоже идем в фоллбэк.
* *Что если у юзера вообще нет ID?* (например, не залогинен). Сразу идем в вариант 3 (показываем популярное).
